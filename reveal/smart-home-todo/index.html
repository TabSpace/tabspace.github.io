获取用户信息【已填】

  • 需要获取昵称，微信uid，手机号
  • 使用 wx.getUserInfo 接口直接弹出授权框的开发方式将逐步不再支持。
  • 官方要求：使用 button 组件，并将 open-type 指定为 getUserInfo 类型，获取用户基本信息。
  • 已授权，则可以继续使用 wx.getUserInfo 接口
  • 未授权，必须呈现 button[open-type="getUserInfo"]

直播流接入【未填】

  • 目前主要直播协议为 hls 与 rtmp
  • hls 支持面广泛，但延迟高，普遍在 10 秒以上，无法控制在 5 秒内
  • rtmp 是 adobe 私有协议，h5 端需要 flash 支持，小程序可直接播放
  • rtmp 直播延迟可控制在 1 秒内
  • 腾讯云提供了对 rtmp 和 hls 的支持

直播流接入

  • 小程序 video 组件默认支持 hls 直播
  • rtmp 实时直播需要启用 live-player 组件
  • 启用 live-player 组件需要配置服务类目为 社交 > 直播
  • 配置该类目需要 信息网络传播视听节目许可证
  • 摄像头可提供推流器来向腾讯云推送 rtmp 上行流
  • 通过腾讯云获取在线直播地址，需要后台开发介入

  • 设备可以触发场景

  • 场景引用了触发另一个场景的设备
  • 引发死循环（物理）

情景模式死循环

  • 方案1：添加熔断，简单易行，但不够友好
  • 方案2：场景创建时进行判断，可提早给予用户提示
  • 每个场景记录触发设备，每个设备记录隶属场景
  • 设备隶属的场景可能有多个，场景会关联多个设备
  • 整个图的遍历可能会是一个高消耗操作

情景模式死循环

  • 可以考虑创建场景记录上游设备列表
  • 继承上游场景的上游设备列表
  • 如关联设备与上游设备列表冲突，则拒绝关联
  • 场景移除设备需要向下游更新场景链
  • 总场景数量应予以控制
  • 与熔断判断相结合

渲染性能优化

  • 减少 mesh 的构建，用 clone 与镜像替代重复构建 mesh

  • 物理引擎是性能杀手，慎用物理引擎与碰撞检测
  • 减少照射同一物体的光源数量，减少光效叠加，控制光源影响的 mesh 数量

光影重叠【未填】

  • 同一物体可照射光源默认仅支持6个
  • 环境光，地面光就占了2个
  • 提高光源支持数量，会大幅牺牲性能
  • 通过指定光源可照射 mesh ，可节约光源数量
  • 影子呈现存在 bug，背后物体被照亮
  • 如何用贴图模拟光影效果，是个大学问

思考与总结

样板间小程序

  • 需要解决持续迭代变更问题
  • 需要完善模型交互，提取SDK
  • 需要完善测试
  • 重点在持续改进用户体验

智家管理平台

  • 前端交互上不存在复杂的坑
  • 主要问题会出在产品逻辑上
  • 设备联网与部署麻烦，需要解决方案
  • 重点在解决适应性，健壮性

3D样板间

  • 坑最多，也最复杂
  • 需要更长时间投入人力踏实技术基础
  • 兼容性问题交给市场与时间
  • 持续迭代，寻找适合的使用场景
  • 未来潜力最大，可形成技术差距
  • 重点在探索实用价值，寻找性能体验平衡点
